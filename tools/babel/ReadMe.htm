The command line options can be entered using a double or a single dash.

Double dash prefix tells Babel you want use long option so that the option must be entered as it is shown in the command line ex.: --help --keyfile etc. Single dash prefix is the short form for an option. The option in this case takes only one character and can be specified in bundle with other option ex.: -not -mpf means disable type obfuscation and enable obfuscation for methods properties and fields.

An option may have aliases. Aliases provide shortcuts for an option: for example --keyfile option has the alias --kf. When Babel parse long options names uses auto abbreviation to resolve the option. So --statistics can be shortening by --stat. Some options are negatable. These options are like switch that can be turned on or off. All negatable options may have the prefix no to turn off or disable the function specified ex.: -not disable type obfuscation.

Babel Rules

Babel rule files are XML files that contain information to guide the process of obfuscation. The rule file consists of a list of rules defined by XML element <Rule></Rule>. This list is enclosed by the element <Rules></Rules> that is also the root document element.

 

<Rules xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"

           version="1.0">

  <Rule name="rule 1" obfuscate="false">

  </Rule>

  <Rule name="rule 2" obfuscate="false">

  </Rule>

  ...

</Rules>

 

The Rules element attribute version is used to define the current schema version and must be set to "1.0". Rules are processed during type obfuscation: each rule defined into the file is checked against the assembly symbols (classes, methods, properties, fields). If a symbol match the entire rule, the symbol obfuscation action is taken according to the Boolean attribute obfuscate value:

 

<Rule name="my rule" obfuscate="false">

  ...

</Rule>

 

The Rule element has another attribute: name that is used to give a friendly name to the rule. The name and obfuscate attributes are mandatory as specified by the XSD schema for rules file. Any Rule element has five other child elements: Access, Target, Pattern, HasAttribute, Description. The first three elements are mandatory, whereas the last two are optional.

 

·         Access: Specify the symbol visibility for this rule. Admitted values are: All or any combination of Public, Protected, Internal, Private.

·         Target: Specify which kind of symbol should be checked.
Admitted values are: All or any combination of Classes, Delegates, Structures, Interfaces, Enums, Events, Methods, Properties, Fields, StaticFields.

·         Pattern: any wildcard expression or regular expression, the Boolean attribute isRegEx states if the pattern is a regular expression. If a regular expression is used, its a good practice to enclose the regular expression definition string in a CDATA section, ex.:

<Pattern isRegEx="true"><![CDATA[^Properties.*]]></Pattern>

 

The optional elements are:

 

·         HasAttribute: specify a list of fully qualified type attributes. If the target has at least one of these attribute the rule match. This element may have the Boolean attribute onEnclosingType. If this attribute is set to "true" the attribute presence is checked on the target symbol enclosing type. Otherwise if it is set to "false" the attribute presence is checked on the target symbol itself.

·         Description: any useful rule description.

                                           

The following example show the content of SafeRules.xml file distributed with Babel package:

 

<Rules xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"

           version="1.0">

  <Rule name="Properties types" obfuscate="false">

    <Access>All</Access>                         

    <Target>All</Target>

    <Pattern isRegEx="false">*Properties*</Pattern>

    <Description>Do not obfuscate all Properties namespace types.</Description>

  </Rule>

  <Rule name="Serializable types" obfuscate="false">

    <Access>All</Access>

    <Target>Fields, StaticFields</Target>

    <Pattern><![CDATA[*]]></Pattern>

    <HasAttribute onEnclosingType="true">System.SerializableAttribute</HasAttribute>

    <Description>Do not obfuscate fields of Serializable types.</Description>

  </Rule>

</Rules>

 

The first rule tells Babel to do not obfuscate any symbol inside the Properties namespace or whatever symbol contains Properties in his fully qualified name. This rule is required by .NET Windows Form applications that put resource access into the Properties namespace. The second rule tells Babel to do not obfuscate fields of types that are declared as serializable. The Serializable attribute presence is checked on the field enclosing type because the XML attribute onEnclosingType is set to true.

How to Add an XML Rule File in Visual Studio Project

XML rule files can be added to Visual Studio project so that Babel can process it during obfuscation.

To Add an XML rule file perform the following steps:

1)   In Visual Studio Solution Explorer, right-click on the project node you want to add the XML rule file

2)   Select from the popup menu Add -> New Item

3)   From the Templates view select XML File and name it: BabelRules.xml

4)   Fill the XML file with the rule definitions

5)   Save and build the project

Obfuscation Agent

The Obfuscation Agent consist of a list of tasks that babel performs against the target assembly type model before the obfuscation process to determinate what are the symbols that should not be obfuscated. The Agent prevents symbols being obfuscated according some common rules:

For example, serializable types name should not be obfuscated. Also Enum types consumed by method that use reflection, such as for instance Enum.GetNames(), should not be obfuscated.

Babel knows a finite set of rules and it perform static code analysis to discover symbol that mach one of these rules. The Agent works to prepare a safer obfuscation process and most of the time the final obfuscated assembly will run as well as the original one. Because the Agent can only perform static code analysis, it may not discover types that are controlled during execution. For example string of type name built at runtime passed to reflection method are opaque to the Agent so these types will be obfuscated and the target will most likely break during execution. For this scenario XML rule files combined with Agent are the recommended solution.

The Agent is enabled by default and it should be used instead of rule files. If you prefer to use an XML rule file, you can run the Agent and use its output to know what types you should add to your rule file.

String Encryption

Babel can encrypt so called User Strings embedded into a managed assembly. During obfuscation babel scan the assembly for all the strings and for each string call an encryption method that is embedded into the target assembly. The original string is then replaced by the encrypted one. At the end of the obfuscation process the encryption metod is deleted from the assembly. During execution the encrypted string is passed to a decryption method to recover the original string value. The Encryption and Decryption method must be embedded into the original assembly and are recognized by their signature or by the attaced attributes EncryptStringMethodAttribute and DecryptStringMethodAttribute declared into the Babel.Code.dll assembly. The encrypt and decrypt methods must be declared as follow:

internal static string EncryptString(string text);

internal static string DecryptString(string text);

If the attributes [EncryptStringMethod] and [DecryptStringMethod] are not used the methods must have the following names: EncryptString, DecryptString. Starting from version 1.4, if encryption and decryption string methods are not provided, Babel can merge into the target assembly a default encrypt and decrypt method and perform as well as string encryption. The default encryption so performed is weak but very fast because it is implemented by a simple XOR algorithm. It is also possible to add the attribute [EncryptStrings(false)] to methods, properties, events and constructors declarations so that all the strings used by these targets will not be encrypted by babel.

Obfuscation of x64 Assemblies

On x64 operating systems, the intermediate language (IL) code of assemblies compiled with "Any CPU" platform is validated by runtime before execution. Because babel during obfuscation inserts invalid IL opcodes (see --invalidopcodes command line option) into any method and property to break tools like Reflector, the obfuscated IL is not valid and the assembly can run only into WOW64 (Windows On Windows 64-bit) subsystem. If the obfuscated assembly is a dll referenced by an executable running in full 64 bit environment, when the executable try to load the obfuscated dll, the application crashes throwing an InvalidProgramException. The only way to generate obfuscated .dll or .exe loaded by 64 bit applications is to disable the injection of invalid opcodes during obfuscation and start babel with the option:

--noinvalidopcodes

(see: Babel Command Line).

So if you want to obfuscate agnostic CPU assembly, remember to add this option to your babel command line.

Integrate Babel Obfuscation in Your Build Process

Facilities are provided for MSBuild technology. To integrate Babel with your build process, edit Visual Studio (.*proj) file to import Bable.Build.targets. This file is located under the MSBuild folder inside the installation directory.

 

Follow the steps required to modify the Visual Studio (.*proj) file:

 

1)   In Visual Studio Solution Explorer, select the project node, right-click and from the context popup menu choose Unload Project command. When the project is unloaded, its icon changes to a folder and the word '(unavailable)' appear next to it.

2)   Right-click the unloaded project and from context popup menu choose Edit.

3)   In the XML project file insert after the line:

 

  <Import Project="$(MSBuildToolsPath)\Microsoft.CSharp.targets" />

 

  the following Import declaration:

 

 <Import Project="<full path to Babel.Build.targets file>" />

 

  for example:

 

 <Import Project="C:\Program Files\Babel\MSBuild\Babel.Build.targets" />

 

4)   Save and reload the project.

 

Rebuild the project and look at the Output panel to see Babel output.

The obfuscated assembly is saved to .\BabelOut subfolder relative to the Output path.

 

The Bable.Build.targets file contains the Obfuscate MSBuild target definition that runs babel.exe passing as the input file the project build output assembly. Bable.Build.targets contains also the property definitions of the command line switches value passed to babel.exe. These properties can be overridden into your (.*proj) file to customize Babel behavior.

For instance, in Bable.Build.targets the property UnicodeNormalization is set to true to enable Unicode normalization by default:

 

<PropertyGroup>

  <UnicodeNormalization>true</UnicodeNormalization>

</PropertyGroup>

 

If you want to disable Unicode normalization into your project, insert into

the .*proj file after the babel import declaration:

 

<Import Project="C:\Program Files\Babel\MSBuild\Babel.Build.targets" />

 

the following lines:

 

<PropertyGroup>

  <UnicodeNormalization>false</UnicodeNormalization>

</PropertyGroup>

 

Other properties are available to override, they are defined into the Bable.Build.targets file as follow:

    <!--

    ============================================================

                         Babel Properties

 

    Several properties that can be overridden in .*proj files.

    ============================================================

    -->

    <PropertyGroup>

        <EnableObfuscation>true</EnableObfuscation>

        <BabelOutputFile></BabelOutputFile>

        <EnableObfuscationAgent>true</EnableObfuscationAgent>

        <GenerateDebug>false</GenerateDebug>

        <BabelRulesFileName>BabelRules.xml</BabelRulesFileName>

        <BableRulesPath Condition="Exists('$(ProjectDir)$(BabelRulesFileName)')">$(ProjectDir)$(BabelRulesFileName)</BableRulesPath>

        <BabelMapOutFile></BabelMapOutFile>

        <FlattenNamespaces>true</FlattenNamespaces>

        <UnicodeNormalization>true</UnicodeNormalization>

        <ObfuscateTypes>true</ObfuscateTypes>

        <ObfuscateEvents>true</ObfuscateEvents>

        <ObfuscateMethods>true</ObfuscateMethods>

        <ObfuscateProperties>true</ObfuscateProperties>

        <ObfuscateFields>true</ObfuscateFields>

        <VirtualFunctions>true</VirtualFunctions>

        <ObfuscateMsil>true</ObfuscateMsil>

        <ILIterations>3</ILIterations>

        <EmitInvalidOpcodes>true</EmitInvalidOpcodes>

        <StringEncryption>false</StringEncryption>

        <SuppressIldasm>true</SuppressIldasm>

        <ShowStatistics>true</ShowStatistics>

        <VerboseLevel>2</VerboseLevel>

    </PropertyGroup>

The EnableObfuscation property can be used to switch on or off Babel obfuscation in particular build configurations.

For example if you want to enable obfuscation in Release build and disable into Debug one, you can add the following lines to your .*proj file:

 

  <Choose>

    <When Condition=" '$(Configuration)|$(Platform)' == 'Debug|AnyCPU' ">

      <PropertyGroup>

        <EnableObfuscation>false</EnableObfuscation>

      </PropertyGroup>

    </When>

    <When Condition=" '$(Configuration)|$(Platform)' == 'Release|AnyCPU' ">

      <PropertyGroup>

        <EnableObfuscation>true</EnableObfuscation>

        <GenerateDebug>false</GenerateDebug>

        <ILIterations>3</ILIterations>

        <StringEncryption>true</StringEncryption>

      </PropertyGroup>

    </When>

  </Choose>

FAQ

Version 1.5.0.0 and earlier:

·         Why I get the Error: Phoenix failed to load a required component: mspdb80.dll, when I use -debug option?
To generate debug .pdb file, babel try to load mspdb80.dll at runtime. This Dll is distributed with Visual Studio and is located into the $Program Files\Visual Studio 9.0\Common7\IDE directory. To sucessuflly generate .pdb file start babel from Phoenix or Visual Studio DOS shell.
